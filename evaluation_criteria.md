## 評価の思想

### 私たちが模倣する「目」

この評価基準は、実際の発表会に参加する審査員の思考回路を再現する。2025夏の発表会に実際に審査員として参加した方々のプロフィール:

- **大企業の事業担当者**（関電工・セブン銀行・三菱地所・ザイマックスデジタル）: 「現場に落ちるか？ 自社のアセットと組み合わせたら面白いか？ この人たちの話をもっと聞きたいか？」
- **メディア**（日本経済新聞社）: 「記事にする価値があるか？ 読者が "へえ" と思うファクトがあるか？ 社会的にどういう意味があるか？」
- **教育関係者**（高専機構・各高専教員）: 「学生はこのプログラムを通じて成長したか？ 思考の枠組みが変わったか？ この先に何が続くか？」
- **スタートアップ関係者**（アルバクロス・GOLDEN EGG Ventures・W4Partners）: 「問題の解像度は高いか？ スケールする可能性はあるか？」

彼らに共通する判断軸:

> **「この人たちは、本当にこの問題を理解していて、それを自分の言葉で伝えられるか？」**

v3.0では「思考の解像度」のみを核に据えていたが、実際の審査では**思考の解像度が「伝わる」こと**が等しく重要だった。深く考えていても伝わらなければ、聞き手の中では「考えていない」のと同じになる。逆に、論理が明快で質問にも的確に答えられるチームは、聞き手に「このチームは分かっている」という確信を与える。

### 「考え抜かれている」とは何か

Peter Thiel は「賛成する人がほとんどいない大切な真実は何か？」と問う。
Y Combinator は「ユーザーと話したか？ 何人に？ 何と言っていたか？」と詰める。

しかし、これらの問いに答えられるだけでは十分ではない。2025夏の審査結果が示したのは:

> **深い思考は、明快な伝達と質疑への対応力によって初めて「見える」ようになる。**

Toku冷stは全員1-2年生のチームだったが、審査員9名の平均でプレゼンテーション4.67/5という最高評価を得た。彼らが最も「考え抜いている」ように見えたのは、思考の深さ自体が飛び抜けていたからではなく、**自分たちが考えたことを過不足なく伝える力**があったからだ。

一方、さるく電街隊は実際にテナント調査や通行者計測を行ったが、質問対応3.44/5と最低評価だった。どれだけフィールドワークをしても、**その知見を問われたときに答えられなければ、聞き手には届かない**。

---

### 明確に評価しないこと

| 評価しない | 理由 |
|-----------|------|
| スライドの装飾・アニメーション・配色 | 視覚的な華やかさは思考の代替にならない |
| 使用技術の高度さそのもの | 手段は目的ではない |
| 作業量・工数 | 頑張りは成果ではない |
| メンバーの学歴・学年 | 属性は能力ではない |
| チーム構成の多様性そのもの | 多様であること自体は成果ではない |

> **重要**: v3.0では「スライドのデザイン・装飾」と一括していたが、v4.0では**装飾のみ**を除外する。論理構造の可視化（データの図示、議論の流れ図等）や、伝達の明快さはむしろ積極的に評価する。これらは「デザイン」ではなく「思考の表現」だからだ。

## 4つの評価観点（各25点・合計100点）

| # | 観点 | 問い | 配点 |
|---|------|------|------|
| 1 | **課題発見の視点** | この問題に気づけたこと自体に価値があるか？ | 25 |
| 2 | **分析の論理性と深さ** | 筋が通っているか？ どこまで掘り下げたか？ | 25 |
| 3 | **伝える力** | 聞き手を納得させられるか？ | 25 |
| 4 | **実行と発展性** | やったか？ その先に何があるか？ | 25 |

> 4観点とも均等配点とする。良い取り組みとはこの4つが全て高い水準で揃っている状態であり、どれか一つが欠けても大きく減点される。

### 実際の審査項目との対応

| 本基準の観点 | 実際の審査項目との関係 |
|------------|---------------------|
| 課題発見の視点 | ← 審査員フォームの「課題発見の視点」に直接対応 |
| 分析の論理性と深さ | ← 「課題発見の視点」の深い部分 + 「質問対応」（本質的理解の証拠） |
| 伝える力 | ← 「プレゼンテーション」+「質問対応」に対応 |
| 実行と発展性 | ← 「プログラム終了後の発展性」に対応 |

## 観点1: 課題発見の視点（25点）

**「この問題に気づけたこと自体に、価値があるか？」**

審査員がまず見るのは「どんな問題を選んだか」だ。良い課題発見には3つの条件がある:
1. **切実さ**: 実際に困っている人がいる
2. **独自の切り口**: 「誰でも言えること」を超えた、このチームならではの視点がある
3. **構造の理解**: なぜその問題が起きているかのメカニズムを把握している

### ルーブリック

| 点数 | 水準 | 具体像 |
|------|------|--------|
| **21-25** | 問題の構造まで理解し、独自の切り口がある | 「誰が」「なぜ」困っているかを数字と構造で説明できる。表面的な症状ではなく根本原因に迫っている。「なぜこの問題は今まで解決されていないのか」にも答えられる。問題の規模感が示されている。自分たちが特にフォーカスする角度が明確。 |
| **16-20** | 良い問題を選んでいるが、深掘りが一段足りない | 問題の存在は明確で、切実さも伝わる。ただし「なぜ？」の因果関係が途中で止まっている、または切り口の独自性がやや弱い。 |
| **11-15** | 問題はあるが、他の誰でも設定できるレベル | 「CO2が多い」「混雑している」等、一般的に知られた課題をそのまま持ってきている。自分たちのフォーカスが示されていない。 |
| **6-10** | 問題が曖昧 | テーマはあるが、具体的に誰が何に困っているのかが見えない。 |
| **1-5** | 問題が見えない | 取り組む問題自体が不明確、または問題ではなく「やりたいこと」からスタートしている。 |

### 自問リスト（採点者用）
- この問題を放置したら、具体的に誰がどう困るのか？ その人数は？
- 「なぜこの問題が存在するのか」の原因構造を説明できているか？
- この問題の中で「自分たちが特に注目する切り口」は何か？ それは独自か？
- 当事者としての実感や現場体験があるか？
- 聞いた瞬間に「ああ、それは確かに問題だ」と思えるか？

### 較正基準（2025夏の実績に基づく）
- Ekochicket（審査員: 課題発見 **4.56**/5 = 最高）: CO2排出と公共交通衰退を二軸で構造化 → **19-21点**相当
- さるく電街隊（審査員: 課題発見 **3.56**/5 = 最低）: 商店街の衰退は一般的に知られた課題 → **13-15点**相当

## 観点2: 分析の論理性と深さ（25点）

**「筋が通っているか？ どこまで掘り下げたか？」**

課題を見つけた後、それをどこまで深く分析したか。ここで測るのは**思考プロセスの論理的一貫性**と**掘り下げの深さ**の両方だ。

「深い分析」の特徴:
- **一次情報を持っている**: 自分の足で得たデータ、インタビュー結果がある
- **因果関係の連鎖がある**: 「AだからB、BだからC」と論理が数珠つなぎになっている
- **数字で語れる**: 感覚ではなく定量データで状況を把握している
- **比較対象がある**: 他の地域・事例と比較して「ここが違う」と言える
- **独自の指標やフレームワークが生まれている**: 既存の分析を超えた知的成果物がある

ただし、分析が深くても**それが論理的に一貫していなければ評価しない**。データを集めただけで「だから何？」に答えられない状態は、深さではなく作業量にすぎない。

### ルーブリック

| 点数 | 水準 | 具体像 |
|------|------|--------|
| **21-25** | 論理が一貫し、独自の知見がある | 一次情報に基づき、「なぜ？」が3回以上重ねられている。課題→分析→洞察→提案の論理チェーンが途切れない。独自の指標・概念が生まれている（例: NLVI、なまらピーク等）。主張の限界や前提も認識している。 |
| **16-20** | 論理は通るが、あと一段深掘りが欲しい | 独自のデータ収集や分析は行っている。部分的に光る洞察がある。ただし論理の一部に飛躍がある、または「なぜ？」があと一層欲しい。 |
| **11-15** | データはあるが、洞察が弱い | 情報収集はしているが、そこから得られた結論が予想通り。分析→提案の間に論理の飛躍がある。 |
| **6-10** | 表面的 | 既存情報のまとめが中心。分析が浅く、論理の組み立てが弱い。 |
| **1-5** | 分析の形跡がない | テーマに対する分析・考察がほぼ存在しない。 |

### 自問リスト（採点者用）
- 課題→分析→洞察→提案の論理チェーンは途切れていないか？
- 「ネットで30分調べれば分かること」を超える情報・分析があるか？
- 自分の足で得た一次情報はあるか？
- 「なぜ？」を何回掘り下げているか？（1回→浅い、3回以上→深い）
- 独自の指標・分類・フレームワークを生み出しているか？
- 「このデータから、だから何が言えるのか？」に明確に答えているか？
- 自分たちの分析の限界・前提を自覚しているか？

### 較正基準（2025夏の実績に基づく）
- なまら★ばらけナイト: 「なまらピーク」独自指標を創出し、観光客の偏りを構造的に分析 → **19-21点**相当
- setomorrow（審査員: 質問対応 **4.33**/5 = 最高）: Q&Aでの的確な応答は論理的理解の証拠 → **16-18点**相当

## 観点3: 伝える力（25点）

**「聞き手を納得させられるか？」**

> **v3.0からの最大の変更点。** v3.0は「スライドのデザインを評価しない」としていたが、2025夏の審査結果は、**伝える力が審査結果を最も大きく左右する要因**であることを示した。Toku冷stがAI評価4位→審査員1位となった主因は、プレゼンテーション力（4.67/5）だった。

ここで評価するのはスライドの美しさではない。**自分たちの思考を過不足なく聞き手に届ける能力**を評価する。

「伝える力」の構成要素:
- **議論の構造が明快**: 聞き手が迷子にならない。「今どこの話をしているか」が常に分かる
- **データの見せ方が適切**: 数字やグラフが主張を裏付けている（飾りではなく証拠として機能している）
- **ストーリーがある**: 問題→気づき→分析→提案→次のステップ、という物語として成立している
- **質問に的確に答えられる**: 想定外の問いにも、自分の言葉で筋の通った回答ができる（暗記ではなく理解の証拠）
- **聞き手の「だから何？」に先回りしている**: 聞き手が疑問を抱く前に答えが提示される

### ルーブリック

| 点数 | 水準 | 具体像 |
|------|------|--------|
| **21-25** | 聞いた後に「なるほど」が残る | 議論の構造が明快で、データが主張を的確に裏付けている。聞き手が自然に結論に導かれる。質問にも自分の言葉で的確に答えられる。「この人たちは本当に分かっている」と確信させる。 |
| **16-20** | 概ね伝わるが、一部不明瞭 | 全体の流れは理解できるが、論理の一部で「なぜそうなるのか？」が残る。データの提示は概ね適切。質問への対応はできるが、深い突っ込みにはやや弱い。 |
| **11-15** | 言いたいことは分かるが、説得力が弱い | 主張は理解できるが、根拠が不十分、または議論の流れに飛躍がある。データの活用が不十分（見せているが主張との接続が弱い）。 |
| **6-10** | 何を言いたいのかが伝わりにくい | 情報の羅列になっている。聞き手が自分で論理を組み立てなければならない。 |
| **1-5** | 伝わらない | 議論の構造がなく、何がポイントなのか分からない。 |

### 自問リスト（採点者用）
- 発表を聞いて（スライドを見て）、3文で要約できるか？ できないなら構造が不明瞭
- データやグラフは「だからこう言える」の根拠として使われているか？ 飾りになっていないか？
- 問題→分析→提案の流れに飛躍はないか？
- 質問に対して、暗記ではなく自分の言葉で答えているか？（理解の深さの最も確実なシグナル）
- 聞き手が「だから何？」と思うタイミングはなかったか？
- 「一番伝えたいこと」が明確に伝わったか？

### 較正基準（2025夏の実績に基づく）
- Toku冷st（審査員: プレゼン **4.67**/5 = 最高、質問対応 **4.22**/5）: 全員1-2年生ながら最も「分かっている」と感じさせた → **21-23点**相当
- さるく電街隊（審査員: プレゼン **4.00**/5 = 最低、質問対応 **3.44**/5 = 最低）: フィールドワークは行ったが、発表と質疑で伝えきれなかった → **11-13点**相当

### スライド・動画のみの評価時の注意

質問対応を直接観察できない場合は、以下をプロキシ（代理指標）として用いる:
- **反論の先回り**: スライドの中で「〇〇ではないか？→実は〇〇」という構造があるか
- **前提条件の開示**: 自分たちの主張の限界を自ら述べているか
- **「なぜその方法か？」への回答**: 代替案との比較や選択理由が示されているか
- **論理の柔軟性**: 初期仮説が修正された形跡があるか（硬直的でない思考の証拠）

## 観点4: 実行と発展性（25点）

**「やったか？ その先に何があるか？」**

> **v3.0からの変更点。** v3.0の「地に足のついた実行」は過去の行動のみを評価していたが、実際の審査員は「発展性」を25%の配点で重視していた。特に学生プロジェクトでは、「今完璧であること」より「この先に何が続くか」が重要。

この観点では2つの軸を評価する:
1. **実行**: 実際にやったこと（フィールドワーク、プロトタイプ、データ収集、実証実験）
2. **発展性**: この取り組みがどこまで広がり得るか（スケーラビリティ、他地域への展開、事業化・政策化の可能性）

ただし、**実行の量だけを評価しない**。v3.0ではフィールドワークの記述を過大評価し、さるく電街隊をAI 2位としたが実際の審査員は5位だった。重要なのは「何を学んだか」であり「何回行ったか」ではない。

### ルーブリック

| 点数 | 水準 | 具体像 |
|------|------|--------|
| **21-25** | 現実から学び、未来が見える | 動くプロトタイプがある、または実証データを得ている。仮説→検証→修正のサイクルを回した形跡がある。「次にやること」が具体的で、聞き手が「一緒にやりたい」と思う。他の地域や文脈への応用可能性が示されている。 |
| **16-20** | 実行があり、発展の方向性も見える | 現地調査やプロトタイプの試行はある。次のステップは見えるが、スケーラビリティや他への応用についてはまだ曖昧。 |
| **11-15** | 調査はしたが、学びと発展が弱い | データ収集はあるが「やってみてどうだったか」が弱い。次のステップが「〇〇したい」レベルの願望にとどまる。 |
| **6-10** | 机上の分析のみ | 二次データの分析が中心で、現場体験がない。将来展望も抽象的。 |
| **1-5** | 実行も展望もない | 考えただけ。現場にも行かず、作ってもいない。次に何をするかも不明。 |

### 自問リスト（採点者用）
- 自分の足で現場に行ったか？ そこから何を学んだか？（回数ではなく学びの質を見る）
- 動くプロトタイプ、試作品、実証データのいずれかがあるか？
- 最初の仮説と最終的な結論は違うか？（修正があれば学んだ証拠）
- 「次に何をするか」の具体性は？（「〇〇したい」→弱い、「来月〇〇する」→強い）
- この取り組みを他の地域や文脈に応用できるか？
- 企業や自治体が「一緒にやりたい」「予算をつけてもいい」と思えるか？
- プログラム終了後もこの活動が続く可能性はあるか？

## 採点の原則

### 「考え抜かれている」が「伝わっている」の見分け方

| 考え抜かれていて伝わっている | 考え抜かれているが伝わっていない | 浅いが上手く見せている |
|---|---|---|
| 数字で語り、その数字の意味を説明する | 数字はあるが、何を意味するか分からない | 印象的な数字を出すが因果関係がない |
| 質問に自分の言葉で答えられる | 質問されると固まる、または暗記を繰り返す | 質問を上手にかわすが中身がない |
| 論理の流れに飛躍がない | 良い分析があるが順序がバラバラ | 流れは綺麗だが根拠が薄い |
| 限界を自ら認めている | 限界は分かっているが言語化できていない | 限界に触れない |
| 仮説が修正された形跡がある | 修正したが説明が不足 | 最初から最後まで同じ主張 |

> **重要**: 「浅いが上手く見せている」は高得点にしない。観点3（伝える力）は高いが観点2（分析の深さ）が低い、という採点になる。全観点の合計で評価するからこそ、本質的な取り組みが報われる。

### 高評価の基準

学生プロジェクトであることを差し引かない。甘くつけることはリスペクトではない。

ただし、**学年や経験の差がある中での「伸びしろ」は観点4（発展性）で拾う**。1-2年生が3軸分析を設計・実行した場合、その成長と今後のポテンシャルは発展性の観点で正当に評価される。

80点以上は「企業の担当者が "次のミーティングを入れたい" 水準」に限定する。

## 評価プロセス

1. **スライド精読**: 各チームのスライドURLから全ページの内容を読み、論理構造を把握する
2. **README参照**: 補助情報としてREADMEを確認する（ただしREADMEの好意的記述を鵜呑みにしない）
3. **4観点の独立採点**: 各観点を独立にルーブリックに照らして採点する
4. **較正チェック**: 2025夏の較正基準と比較し、スケールの一貫性を確認する
5. **理由の記述**: 各観点の点数の根拠を1-2文で明記する
6. **総評**: 「この取り組みは社会にどんな意味があるか」を2-3文で記述
7. **CSV統合出力**: 全チーム分を統合

### スライドが読めない場合の注意

READMEのみで評価する場合、以下のバイアスに注意する（v3.0の教訓）:

| バイアス | 具体例 | 対策 |
|---------|--------|------|
| **README記述の好意的バイアス** | 運営者がフィールドワークを好意的に記述→実際の質は不明 | READMEの「やった」記述を額面通りに受け取らない。「何を学んだか」が書かれていない場合は減点 |
| **伝える力の不可視** | READMEからはプレゼン力・Q&A力が全く見えない | 観点3は保守的に採点し、幅を持たせる（例: 13-17点と幅を示す） |
| **概要の長さバイアス** | 運営者の記述量がチームごとに異なる | 記述量ではなく、記述内容の具体性で判断する |
